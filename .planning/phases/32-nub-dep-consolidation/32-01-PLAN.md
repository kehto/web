<objective>
Atomic single-pass migration: swap every `@kehto/*` + demo napplet package manifest from 8 split `@napplet/nub-<domain>@^0.2.1` peer/dev deps to the consolidated `@napplet/nub@^0.2.1` package, rewrite every in-repo TS import AND every in-repo comment substring referencing `@napplet/nub-<domain>` to the subpath form, and regenerate `pnpm-lock.yaml` so the dual-instance pitfall is structurally eliminated across kehto's importer blocks.

Purpose: DEP-01..03 are three faces of one change — manifests, imports, and the kehto-owned importer blocks of the lockfile all describe the same dependency graph. Splitting them across multiple plans would land the workspace in an intermediate broken state (manifest says one thing, imports say another, lockfile says a third). One atomic plan, one commit, one green type-check/build.

Lockfile scoping note: pnpm-lock v9 has two top-level sections — `importers:` (per-workspace-package dep declarations) and `packages:` (flat resolution graph including transitives). `@napplet/sdk@0.2.1` and `@napplet/shim@0.2.1` both still declare all 9 split `@napplet/nub-*` packages as runtime deps. Every demo napplet pulls `@napplet/sdk` / `@napplet/shim` directly, so the lockfile's `packages:` section will continue to contain split-dep lines until @napplet upstream drops them (tracked against @napplet, not kehto). The migration target is kehto's **importer blocks** only — the 4 `@kehto/*` packages plus the 1 in-repo demo napplet (`media-controller`) whose manifest this plan owns. Anti-term greps must be scoped accordingly (see acceptance criteria).

Comment-prose rewrite note: the anti-term grep is substring-based, not AST-aware. Any JSDoc / line comment containing the literal string `@napplet/nub-<domain>` will trip the grep even though it's prose, not a code import. This is a grep-narrowness consequence, not scope creep — 32-CONTEXT.md Scope-in covers any file whose contents reference the split packages, and the success criterion is "zero source-file substring matches", which includes comments.
