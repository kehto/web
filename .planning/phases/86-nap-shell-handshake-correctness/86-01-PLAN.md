<objective>
Bring the NAP-SHELL `shell.init` / `shell.ready` handshake into conformance with the merged NAP-SHELL spec via two surgical corrections, both isolated to the shell-side `shell-ready.ts` `shell.init` build path:

- **SHELL-01 (gap G1):** `shell.init` MUST be sent exactly once per napplet lifecycle. Today `handleShellReady` calls `postShellInit` unconditionally on every `shell.ready`; a duplicate `shell.ready` resends `shell.init` (MUST violation). Add a per-windowId "init already sent" guard.
- **SHELL-02 (gap G2):** the `class` field in the `shell.init` wire payload MUST be `number | null` (opaque integer class code), not the internal string label (`'class-1'`/`'class-2'`). Map the internal `NappletClass` label to a numeric wire code ONLY at the point of building the `shell.init` payload.

Purpose: real protocol-conformance bug (G1 HIGH severity) + wire-type drift (G2 MEDIUM) blocking NAP-SHELL conformance for milestone v1.21.

Output: a guarded, spec-conformant `shell.init` emission; regression tests; a `@kehto/shell` changeset.
</objective>
