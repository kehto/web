# Plan 93-01: Dev Runtime Simulation Controls

## Goal

Expose a typed environment simulation contract for the single-window dev runtime so napplet authors can exercise common host modes from CLI flags or a config file without writing custom host code or opening the full playground.

## Requirements

- SIM-01: Accept sensible options for capability toggles, ACL, firewall, identity/signer mode, relay fixtures, storage persistence, cache behavior, upload backend, media/audio behavior, config defaults, and theme defaults.
- SIM-02: CLI flags and config files share one schema; invalid combinations fail before serving the runtime with actionable messages.
- SIM-03: The minimal UI exposes compact simulation state and small development-only adjustment affordances without adding playground panes.
- SIM-04: Defaults provide useful real behavior across the wired services for a local app with no custom config.

## Implementation Steps

1. Add a `DevRuntimeSimulation` schema and normalizer in `@kehto/dev-runtime`, with defaults for identity, relay, storage, cache, upload, media, notifications, config, theme, ACL, firewall, intent, and cvm modes.
2. Extend `DevRuntimeRawOptions`, `DevRuntimeOptions`, and `DevRuntimeHostConfig` to carry normalized simulation settings.
3. Teach the CLI to read a JSON config file, merge it with CLI flags, and expose focused flags for common simulations such as identity mode, relay mode, storage mode, upload mode, media/notification toggles, theme mode, and config values.
4. Wire the browser host service factories to the simulation settings so disabled modes are explicit, fixed identity values are deterministic, config/theme defaults flow through service responses, and runtime state reports the active modes.
5. Add compact top/bottom bar controls for development-only toggles that can safely adjust runtime simulation state at reload boundaries without becoming a panel UI.
6. Add unit tests for schema defaults, CLI/config merge precedence, validation failures, and served host config JSON.
7. Extend focused Playwright coverage to prove visible simulation status, reload-bound adjustment, and representative service behavior under non-default settings.
8. Update docs and planning state, then run focused package checks plus root gates that are practical before committing.

## Risks and Constraints

- Keep the iframe HMR model unchanged: target app URL stays the source of truth.
- Keep visible chrome to one top bar and one bottom bar.
- Do not add dependencies; use a small explicit JSON schema normalizer.
- Do not invent unsupported production behavior. Development adapters may be deterministic, but unsupported or disabled modes must answer clearly.
- Keep config schema stable enough for docs and API consumers; reject ambiguous aliases instead of silently accepting typos.

## Success Criteria

- Config file and CLI flags normalize into one `DevRuntimeSimulation` object.
- Invalid simulation combinations fail before the server starts.
- The host UI shows compact simulation status and can adjust safe development modes without adding side panels.
- Browser-host service responses reflect simulation settings for identity, relay/storage/upload/media/notify/config/theme paths.
- Unit and focused e2e evidence prove defaults and at least one non-default simulation profile.
