# 97-01 Plan: NAP-LISTS Runtime Parity

## Objective

Add `lists.*` support end to end in Kehto while preserving the one-NAP milestone boundary.

## Tasks

1. Runtime: register `lists` in NAP dispatch and service-only handlers.
2. Shell: add `ListsHooks`, advertise `lists` only when `hooks.lists?.isAvailable()` returns true, and extend shell-init tests.
3. Services: add `createListsService` with host hooks for `supported`, `add`, and `remove`; return upstream-compatible envelopes for success, unsupported hooks, thrown hooks, and unknown methods.
4. Paja: wire deterministic in-memory lists behavior and update parity metadata from deferred to advertised.
5. Playground: add deterministic demo hook/service, a `lists-demo` napplet, demo registration, gateway artifacts, and Playwright coverage.
6. Release readiness: add changesets for output-changing packages and run focused plus full gates.

## Must Haves

- LISTS-01: `lists` appears in `domains` and `naps` only when wired; disabled-domain override removes it.
- LISTS-02: `lists.*` dispatches through the runtime service registry.
- LISTS-03: service result shapes match upstream `@napplet/nap/lists` contracts.
- LISTS-04: Paja advertises and handles `lists` without new chrome.
- LISTS-05: playground demo proves supported/add/remove through shell path.
- LISTS-06: build/type/unit/e2e/docs/slop/diff gates pass before PR.

## Risks

- Playground napplet count guards can fail if demo registration and gateway expected counts are not updated together.
- Paja/browser-host build can serve stale `dist`; rebuild `@kehto/paja` before browser proof.
- `@napplet/core` list types are peer-owned; do not duplicate wire semantics beyond the service hook boundary.
