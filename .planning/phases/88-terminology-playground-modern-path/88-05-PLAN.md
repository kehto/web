<objective>
Make `@kehto/shell`'s emitted `shell.init` capabilities conformant with the released
`@napplet/shim@0.13` / `@napplet/core@0.12` NAP-SHELL contract. 0.13 reads
`capabilities.{ domains, protocols }` (verified: feeding kehto's `{naps,nubs,sandbox}` to the
real 0.13 `makeSupports` returns false for everything). Add the conformant `{ domains, protocols }`
shape as a SUPERSET — keep `naps`/`nubs`/`sandbox` for older-shim back-compat (TERM-05).
</objective>

## Contract (verified — @napplet/core@0.12)
```
interface ShellCapabilities { domains: string[]; protocols: Record<string, string[]> }
interface ShellInitMessage { type:'shell.init'; capabilities: ShellCapabilities; services: string[]; class: number|null }
supports('relay')        // true iff 'relay' ∈ capabilities.domains
supports('inc','NAP-2')  // true iff capabilities.protocols['inc'] includes 'NAP-2'
```
Permissions: confirm how the 0.13 shim handles `supports('perm:<x>')` (read the installed shim dist
`node_modules/.pnpm/@napplet+shim@0.13.0/.../dist/index.js` + `@napplet/nap@0.12` shell `makeSupports`).
Map kehto's sandbox permissions into whatever the 0.13 contract expects (likely `perm:<x>` entries in
`domains`, or a passthrough). Keep kehto's default-empty sandbox behavior intact.

## Mapping (from existing buildShellCapabilities data)
- `domains` = the bare NAP domains kehto advertises (the current `naps` entries MINUS the
  `inc:NAP-NN` protocol strings): relay/outbox/identity/storage/inc/theme/keys/media/notify/config/
  resource/connect/class/cvm + conditional upload/intent. Plus any `perm:<x>` entries if that's how
  0.13 expects permissions.
- `protocols` = `{ inc: ['NAP-01','NAP-02','NAP-03','NAP-04','NAP-05','NAP-06'] }` (derive from the
  existing NAP_INC_PROTOCOLS — strip the `inc:` prefix into the per-domain list).
- KEEP `naps`, `nubs`, `sandbox` exactly as today (superset; do not remove — TERM-05).
