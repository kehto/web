<objective>
Switch the playground to the modern `naps`/`inc` capability path and migrate the 4 legacy napplets' capability declarations from `ifc` to `inc` — atomically. Reading `naps` makes `supports('ifc')` return false (naps lacks `ifc`), so the bot/chat/feed/profile-viewer napplets MUST migrate their capability declarations in the same change or their boot-time capability checks fail. The `ifc.*` wire (SDK imports + `ifcOn`/`ifcEmit`) stays untouched — `@napplet/nub@0.5.0` has no `./inc` subpath and the runtime dual-routes `ifc.*`/`inc.*`.

Purpose: TERM-02 + TERM-03 (G6) — exercise the path the real shim uses. TERM-05 guard: dual-emit and existing capability-payload assertions stay green.
Output: Updated shared-vite-config, demo-hooks, 4 napplet configs+mains, and the gateway-artifact-parity capability assertions.
</objective>

  naps: string[]    // PRIMARY — bare domains + `inc` + `inc:NAP-01..06`. Lacks `ifc`.

  supports('inc') === true,  supports('inc:NAP-01') === true

  All bare domains used by the 9 napplets (relay, storage, theme, identity, ...) are in BOTH arrays so they are unaffected.

@napplet/nub@0.5.0 exposes ONLY `./ifc/*` subpaths. The `import { ifcOn, ifcEmit } from '@napplet/nub/ifc/sdk'` lines and the `ifc.*` wire messages MUST stay `ifc` — there is no `./inc` SDK; the build breaks if changed and the wire must remain `ifc.*` (runtime dual-routes to the same handler).
