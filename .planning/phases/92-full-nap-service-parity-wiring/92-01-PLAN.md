# Plan 92-01: Dev Runtime NAP and Service Parity

## Goal

Replace the Phase 91 dev-runtime-only `shell.init` bootstrap with real Kehto shell/runtime wiring and add a static guard that compares the advertised dev-runtime surface against the current local `@napplet/nap` web domains.

## Requirements

- PARITY-01: Advertise every current web NAP domain from `/home/sandwich/Develop/napplet/packages/nap/src`: `shell`, `relay`, `outbox`, `storage`, `identity`, `keys`, `config`, `resource`, `theme`, `notify`, `media`, `upload`, `intent`, `cvm`, `inc`, plus deprecated `ifc` compatibility.
- PARITY-02: Wire available Kehto services through real implementations or deterministic development adapters: relay/outbox, storage, identity, keys, config, resource, theme, notify, media, upload, intent, cvm, ACL, firewall, NIP-5D identity/session state, manifest cache, and artifact/cache behavior.
- PARITY-03: Fill package-level gaps instead of hiding missing support behind dev-runtime shims; document intentional compatibility-only domains in tests.
- PARITY-04: Add a static parity guard so new `@napplet/nap` domains cannot silently drift from Kehto/dev-runtime support.

## Implementation Steps

1. Introduce a browser-side dev runtime adapter that creates `ShellBridge` with local development hooks and service factories instead of manually posting `shell.init`.
2. Register the single iframe with `originRegistry` and the runtime session registry before target navigation and after reloads, using the configured dev window identity.
3. Wire deterministic default services for the dev runtime:
   - relay/outbox with an in-memory relay pool/router
   - storage through existing runtime localStorage persistence
   - identity, keys, config, resource, theme, notify, media, upload, intent, and cvm via `@kehto/services`
   - ACL/firewall/manifest cache through the runtime/shell adapters
4. Export a single capability-domain source for dev-runtime parity tests; do not leave a second hand-maintained browser-host domain list.
5. Add unit/static tests:
   - dev-runtime capabilities include the Phase 92 domain surface and structured `protocols.inc`
   - local `@napplet/nap` web domain directories are covered, with `ifc` documented as compatibility and `shell` as handshake domain
   - advertised `upload` and `intent` require the dev adapters to be wired
6. Extend focused e2e so the hosted fixture sends representative NAP traffic through the real bridge and receives results for at least storage, config, theme, notify, identity, upload, intent, and cvm, while retaining the one-iframe reload proof.
7. Update docs and planning state, then run focused package tests, focused e2e, build/type/unit/docs/slop gates.

## Risks and Constraints

- Preserve framework-agnostic HMR: iframe `src` remains the target app URL; no framework dev-server coupling.
- Keep default chrome minimal: top bar, bottom bar, one iframe.
- Avoid new dependencies.
- `@napplet/nap` is an adjacent local source checkout, not a workspace dependency; the parity guard may read its source tree in tests but package code must not import it by relative path.
- If a service cannot be made meaningful in this phase, the test must name the unsupported behavior explicitly instead of silently advertising it.
