<demo_runtime_behaviors>
<!-- Key behaviors to account for in spec design: -->

1. **relay.publish in the demo stubs out**: The demo's relayPool (apps/demo/src/shell-host.ts:395-408) does not implement `isAvailable()`, so `hooks.relayPool?.isAvailable()` returns undefined/falsy. The runtime's `handleRelayMessage` case 'publish' (packages/runtime/src/runtime.ts:576-604) falls through to reply `{ type: 'relay.publish.result', id, accepted: false, message: 'no relay pool available' }`. The SDK's publish wrapper resolves with `undefined` (no `result.error` so no reject), so composer's success branch runs and `eventId = 'unknown'` → status becomes `'published: unknown'`. The debugger DOES show the `relay.publish` envelope type string regardless.

2. **relay.publishEncrypted requires a signer**: `hooks.auth.getSigner()` (apps/demo/src/shell-host.ts:416-419) returns the demo host signer from signer-connection. If no signer is connected (fresh demo load has no connected signer by default per Phase 17), `peSigner` is null and the runtime replies `{ type: 'relay.publishEncrypted.result', ok: false, error: 'no signer configured' }`. The SDK wrapper sees `result.error` and rejects — composer's catch branch runs, status becomes `'denied: no signer configured'`. The debugger DOES show the `relay.publishEncrypted` envelope type string regardless.

3. **storage.setItem/getItem**: packages/shell/src/hooks-adapter.ts:256-285 statePersistence is localStorage-backed. Keys are scoped per napplet identity (via the nub-storage handler's scope prefix). Values survive `page.reload()` because reload does NOT clear localStorage. Values do NOT survive `page.goto('/')` following a `localStorage.clear()` (which is what demoBeforeEach does at the start).

4. **notify.create from the toaster**: Plan 19-04 dual-registered notification-service under 'notify'. The runtime's handleNotifyMessage routes `notify.create` to that handler which generates a shell-side id, adds the notification to its per-window list, fires `onChange` (which routes to notification-demo.ts which fires a toast), and replies `{ type: 'notify.created', id }`. The toaster's listener appends an <li> with that id to #toaster-list. The host toast layer receives the Notification object via the controller's subscribe callback and renders a .notif-toast in #notification-toast-layer.
</demo_runtime_behaviors>
