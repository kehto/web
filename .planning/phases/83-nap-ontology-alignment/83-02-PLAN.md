<objective>
Make the INC/IFC dispatch rail vocabulary-aware (D4 + D6 / ALIGN-05): register
the existing IFC handler ALSO under dispatch key `inc`, and stop the IFC handler
from hardcoding the `ifc.` prefix on every outgoing message so a `>=0.9.0`
napplet (which only listens for `inc.*`) works end-to-end.

Purpose: Core 0.5.0 `createDispatch` routes `${domain}.*` by raw string prefix
and accepts any key, so `inc.*` messages reach the handler once registered — but
the handler currently emits `ifc.event` / `ifc.subscribe.result` etc. that a
0.9.0 napplet never listens for. The handler must echo the requester's prefix on
direct responses and use each recipient's own prefix on pushes.
Output: dual-registered dispatch key, domain-aware ifc-handler, and `inc.*`
roundtrip + mixed-vocabulary tests; all existing `ifc.*` tests stay green.
</objective>
