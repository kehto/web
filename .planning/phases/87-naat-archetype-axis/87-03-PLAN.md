<objective>
ARCH-03: get the archetype axis into the playground. Two pieces:

1. Tag a real playground napplet. Extend `definePlaygroundNappletConfig` with an optional,
   validated `archetypes` option, plumb it into `recomputeManifest` so the locally re-signed
   NIP-5D dev manifest emits `["archetype","<slug>","<NAP-N>"]` tag(s). Tag `profile-viewer`
   with `["archetype","profile","NAP-1"]` (profile/identity focused; NAP-1 is the profile topic family).

2. Populate the playground intent catalog from RESOLVED manifests via the ARCH-02 adapter.
   Carry `archetypes` (and `title`) through `napplet-resolver.ts`'s `PlaygroundNapplet`, then add
   a small `buildPlaygroundIntentCatalog(napplets)` that maps resolved napplets through
   `manifestToIntentCatalogEntry` from `@kehto/services`. A full interactive intent UI is OPTIONAL
   and OUT OF SCOPE — the requirement is only that the catalog is sourced from resolved manifests
   (provable here by a unit test; the end-to-end dispatch proof lives in 87-02's integration test).

Purpose: closes G3's "host-injected catalog" gap in the actual playground — availability is now
derivable from signed manifest tags, not hardcoded host data.

Output: archetypes plumbing in shared-vite-config + profile-viewer tag + PlaygroundNapplet field
+ catalog builder + its unit test. No changeset (playground is private; shell types unchanged).
</objective>
