<class_invariant_extension_notes>
The existing Phase 38 class-invariant.spec.ts exercises `class-2 theme-switcher relay:write denied` uniformly across 8 NUB domains via `__injectNubEnvelopeAsNapplet__`. The extension MUST preserve that exact trigger pattern — the invariant is "the same relay:write denial fires regardless of which NUB domain issued the request." Phase 40 simply adds config + resource to the ACTIVE_NUB_DOMAINS array and the NubDomain union; the injected envelope type stays `relay.publish` (NOT `config.get` or `resource.bytes`) because the cross-NUB invariant is about the GATE's uniformity, not the injected domain's semantics.

Alternative design (NOT taken, flagged to prevent confusion): we could inject `config.get` or `resource.bytes` envelopes to prove that config/resource's sender caps are ALSO gated by class — but the existing spec's comment block explicitly documents that the test asserts `relay:write` uniformly. Keeping the extension scoped to "more parameterized test labels on the same trigger" preserves Phase 38's design intent.
</class_invariant_extension_notes>
