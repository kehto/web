  File: `packages/shell/src/shell-bridge.ts`, Function: `createShellBridge` -> `handleMessage`, Line: 155 (verify against actual source).

  File: `packages/runtime/src/runtime.ts`, Function: `handleMessage`, Line: 1005 (verify).
  Code: `if (!Array.isArray(msg) || msg.length < 2) return;`
  What fails: Even if an envelope bypasses shell-bridge, runtime's own guard drops it.
  Reproduction: Call `runtime.handleMessage(windowId, { type: "relay.subscribe", ... } as any)`. Returns immediately.

- **Failure Point 3: AUTH Queue — Messages Queued Forever**
  File: `packages/runtime/src/runtime.ts`, Function: `handleMessage`, Lines: 1010-1014 (verify).
  Code: the `if (!sessionRegistry.getPubkey(windowId))` block that pushes to `pendingAuthQueue`.
  What fails: NIP-5D napplets never send REGISTER/AUTH, so `sessionRegistry.pubkey(windowId)` is never set. All messages are queued in `pendingAuthQueue` indefinitely.
  Reproduction: Load a napplet using `@napplet/shim` v0.2.0. Send any NUB message. Check `pendingAuthQueue.size` — grows indefinitely.
  Impact: CRITICAL — memory leak + complete message loss for NIP-5D sessions.

- **Failure Point 4: enforce.ts Unknown Verb Fallback**
  File: `packages/runtime/src/enforce.ts`, Function: `resolveCapabilities`, Lines: 99-102 (verify).
  Code: `default: return { senderCap: 'relay:write', recipientCap: null };`
  What fails: If an envelope somehow passes both guards, `msg[0]` is `undefined`. Falls to default case requiring `relay:write`. A `relay.subscribe` (which needs `relay:read`) is checked against `relay:write`. Napplets with only `relay:read` capability are denied.
  Reproduction: Pass `[{ type: "relay.subscribe", ... }]` (wrapped in array) to `resolveCapabilities`. Returns `{ senderCap: 'relay:write' }` instead of `{ senderCap: 'relay:read' }`.
  Impact: HIGH — wrong capability enforced, legitimate requests denied.

- **Failure Point 5: state-handler.ts Topic Routing**
  File: `packages/runtime/src/state-handler.ts`, Function: `handleStateRequest`, Lines: 82-84 (verify).
  Code: The `event.tags?.find((t) => t[0] === 't')` extraction.
  What fails: `storage.get` envelope objects are never recognized as `BusKind.IPC_PEER` events, so this handler is never invoked. `window.napplet.storage.getItem()` hangs until timeout.
  Reproduction: With `@napplet/shim` v0.2.0: `await window.napplet.storage.getItem('key')` — hangs until shim timeout (5-30 seconds).
  Impact: HIGH — storage API completely non-functional for NIP-5D napplets.

- **Failure Point 6: service-dispatch.ts Topic-Prefix Routing**
  File: `packages/runtime/src/service-dispatch.ts`, Function: `routeServiceMessage`, Lines: 39-44 (verify).

  What fails: `ifc.emit` envelope objects never produce an IPC_PEER event with a `t` tag. Even if invoked, `ifc.emit` uses `.` not `:`, so `colonIndex === -1` returns false. All NUB-format service messages are unrouted.
  Reproduction: Register an audio service handler. Send `{ type: "ifc.emit", topic: "audio:play", ... }` from updated shim. `routeServiceMessage` is never invoked. Audio handler never fires.
  Impact: HIGH — all service handlers (audio, notifications) unreachable via NIP-5D messages.

    ```typescript
    interface Identity {
      readonly pubkey: string;   // AUTH keypair pubkey — CHANGES
      readonly dTag: string;
      readonly hash: string;
    }
    // Composite key: pubkey:dTag:hash
    function toKey(identity: Identity): string {
      return `${identity.pubkey}:${identity.dTag}:${identity.hash}`;
    }
    ```
  - Target identity contract (after migration):
    ```typescript
    interface Identity {
      readonly pubkey: string;   // DEPRECATED — becomes windowId or empty string

    // pubkey field kept for backward compat during data migration only
    ```
