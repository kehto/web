<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| napplet → runtime choke point | Untrusted envelopes; the firewall gate runs AFTER ACL, before dispatch (defense in depth) |
| napplet-reported state → firewall | The napplet must NEVER be able to relax its own limits by self-reporting focus |
| consent UX → policy persistence | The user's remembered choice must be bounded (persisted once), not re-promptable on demand |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-81-03 | Information Disclosure | firewall reject error string | mitigate | Error envelope carries only `'firewall: ' + result.reason` (human-readable) — no stack, no config internals, no counter state leaked to the napplet (FOCUS of security brief item 1). |
| T-81-FOCUS | Spoofing | focus context | mitigate | Focus comes ONLY from `getFocusContext(windowId)`; absent → `{ focused: true }` (relax-only, cannot tighten via forgery). Napplet self-reported interaction is never read (FOCUS-01, security brief item 2). |
| T-81-04 | Denial of Service | consent prompt spam | mitigate | On `ask` resolve, the choice persists as a per-napplet policy via `firewallPersistence`; subsequent messages match the remembered policy and never re-prompt — a malicious napplet cannot spam unbounded prompts (POLICY-02, security brief item 3). |
| T-81-05 | Elevation of Privilege | bypassing reject | accept | The gate `return`s on `'drop'` before `dispatchNubEnvelope`; there is no path from reject to dispatch (verified by integration tests in Plan 03). |
</threat_model>
