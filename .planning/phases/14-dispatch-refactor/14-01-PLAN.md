```typescript
export type NubHandler = (message: NappletMessage) => void;

export interface NubDispatch {
  /** Register a NUB domain handler. Throws if the domain is already registered. */
  registerNub: (domain: string, handler: NubHandler) => void;
  /** Dispatch a message to the handler matching its domain prefix. Returns `true` if handled. */
  dispatch: (message: NappletMessage) => boolean;
  /** Return all currently registered domain strings. */
  getRegisteredDomains: () => string[];
}

export function createDispatch(): NubDispatch;
```

Key semantics (from @napplet/core dispatch() source):
- `dispatch(message)` extracts domain by splitting `message.type` on first `.`
- Returns `true` if handler exists and was called
- Returns `false` if: no dot in type, empty domain, or no handler registered for domain
- `registerNub()` THROWS if the same domain is registered twice — register each domain exactly once

Adapter bridge (NubHandler expects only NappletMessage; kehto handlers need windowId):
windowId is NOT part of NappletMessage — it lives in the runtime scope (handleMessage's `windowId` parameter). The adapter must close over `windowId` at dispatch time. Since registerNub() happens once at init but dispatch() happens per-message with a different windowId each time, windowId MUST be captured via closure at the dispatch() call site, NOT at registerNub() time.

**CRITICAL pattern** — windowId-carrying closure:
Option A (preferred — re-register per handleMessage call is WRONG because registerNub throws on duplicate):
Option B (preferred — create adapters once that read windowId from a runtime-scoped variable):
Option C (preferred — stash windowId on the envelope as a non-enumerable field, adapter reads it off):

Use **Option B** — a single runtime-scoped `currentWindowId: string | null` variable set inside handleMessage() before calling dispatch(), cleared in a try/finally. Each named adapter function reads that variable and calls its kehto handler:

```typescript
// At runtime init (inside createRuntime):
let currentWindowId: string | null = null;
const nubDispatch = createDispatch();

const relayAdapter: NubHandler = (msg) => {
  if (currentWindowId === null) return;
  handleRelayMessage(currentWindowId, msg);
};
const identityAdapter: NubHandler = (msg) => {
  if (currentWindowId === null) return;
  handleIdentityMessage(currentWindowId, msg);
};
// ... 6 more named adapters in registration order: relay, identity, keys, media, notify, storage, ifc, theme

nubDispatch.registerNub('relay',    relayAdapter);
nubDispatch.registerNub('identity', identityAdapter);
nubDispatch.registerNub('keys',     keysAdapter);
nubDispatch.registerNub('media',    mediaAdapter);
nubDispatch.registerNub('notify',   notifyAdapter);
nubDispatch.registerNub('storage',  storageAdapter);
nubDispatch.registerNub('ifc',      ifcAdapter);
nubDispatch.registerNub('theme',    themeAdapter);

// Inside handleMessage(), replacing the switch (runtime.ts:1078-1088):
currentWindowId = windowId;
try {
  nubDispatch.dispatch(envelope);
  // If dispatch returns false, unknown domain — silent drop per NIP-5D spec. No fallthrough emission.
} finally {
  currentWindowId = null;
}
```

Rationale for Option B over alternatives:
- Option A (re-register every message) fails because registerNub() throws on duplicate domain.
- Option C (stuffing windowId on the envelope) mutates message shape, which could leak into downstream serialization paths and is more surprising than a scoped variable.
- Option B matches kehto's existing closed-over state pattern (serviceRegistry, aclState, sessionRegistry are all closed over the same way).
- Named adapters (not inline arrows) preserve readable stack traces per 14-CONTEXT.md.
