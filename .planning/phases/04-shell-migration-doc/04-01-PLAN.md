**Section 1: Envelope Guard Update (SH-01)**

Write a complete section covering:

1.1 **Background** -- The ShellBridge is a thin adapter that receives raw MessageEvents from window.addEventListener('message'), maps event.source to a windowId via originRegistry, and passes the message to runtime.handleMessage(). The critical gate is at shell-bridge.ts line 155: `if (!Array.isArray(msg) || msg.length < 2) return;`. This is GAP-ANALYSIS.md Failure Point 1 -- it drops 100% of NIP-5D envelope traffic. Explain why this is the FIRST code that must change: the shell-bridge guard fires before the runtime's own guard (runtime.ts:1005), so fixing runtime dispatch alone does nothing if the shell blocks all envelopes.

1.2 **Old Guard vs New Guard** -- Side-by-side comparison:
- Old: `if (!Array.isArray(msg) || msg.length < 2) return;` (line 155) -- only accepts NIP-01 arrays with 2+ elements
- New: Envelope-first check per GAP-ANALYSIS.md section 5.3 boundary contract:
  ```typescript
  // NIP-5D envelope objects (new path)
  if (typeof msg === 'object' && msg !== null && typeof msg.type === 'string') {
    runtime.handleMessage(windowId, msg);
    return;
  }
  // Legacy NIP-01 arrays (backward compat)
  if (Array.isArray(msg) && msg.length >= 2) {
    runtime.handleMessage(windowId, msg);
    return;
  }
  // All else: silently drop
  ```
- Explain why envelope check comes FIRST: NIP-5D napplets are the primary target; legacy napplets are the backward-compat fallback. The order also prevents an array-of-objects from being accidentally treated as a NIP-01 message.

1.3 **sendToNapplet Signature Change** -- Document the outbound side:
- Current: `sendToNapplet: (windowId: string, msg: unknown[]) => void` in hooks-adapter.ts line 94
- Target: `sendToNapplet: (windowId: string, msg: NappletMessage | unknown[]) => void`
- The implementation (`win.postMessage(msg, '*')`) does not change -- postMessage accepts any structured-cloneable value. The signature change is for TypeScript type safety only.
- Reference the RuntimeAdapter.sendToNapplet type from RUNTIME-MIGRATION.md section 4.7

1.4 **ShellBridge Interface Changes** -- Document changes to the ShellBridge public interface:
- `sendChallenge(windowId)` -- REMOVE (AUTH eliminated, per RUNTIME-MIGRATION.md section 2)

- `handleMessage(event)` -- Guard change documented in 1.2
- `registerConsentHandler(handler)` -- UNCHANGED (consent gating still needed for destructive signer operations)

**Section 3: Capability Advertisement Design (SH-03)**

Write a complete section covering:

3.1 **Background** -- NIP-5D section "Runtime Capability Query" (lines 72-84) specifies two APIs:
- `window.napplet.shell.supports(name)` -- boolean, checks NUB capability OR sandbox permission
- `window.napplet.services.has(name)` -- boolean, checks service availability
The current shim (index.ts line 351) stubs `shell.supports()` to always return `false`. This section documents the shell-side design that makes it functional.

3.2 **Current State** -- The shim installs `window.napplet.shell.supports()` as a synchronous function returning `false`. The shim comment says "TODO: Shell populates supported capabilities at iframe creation". The mechanism for shell population is undefined -- the shim has no postMessage round-trip for capability queries (unlike relay, signer, storage which use envelope messages).

3.3 **Design: Shell-Populated Capability Set** -- Document the recommended mechanism:
- At iframe creation, the shell determines which NUBs it supports (e.g., relay, signer, storage, ifc). This is known statically from the shell's configuration -- it is the set of NUB handlers registered with the runtime.
- The shell also determines sandbox permissions (popups, modals, forms, downloads) from the iframe's sandbox attribute.
- This capability set is injected into the iframe context alongside window.nostr (see Section 2). The shim reads it from a shell-provided data structure rather than hardcoding `false`.
- Show the data shape:
  ```typescript
  interface ShellCapabilities {
    nubs: string[];       // e.g., ['relay', 'signer', 'storage', 'ifc']
    sandbox: string[];    // e.g., ['popups', 'modals'] — derived from sandbox attribute tokens
  }
  ```
- `shell.supports(name)` checks `nubs.includes(name) || sandbox.includes(name)`

3.4 **Interaction with NIP-5A Manifest requires Tags** -- Document the relationship:
- Manifest `requires` tags declare what the napplet needs: `["requires", "relay"]`, `["requires", "signer"]`
- At load time, the shell checks manifest requires against its own capability set
- If a required NUB is absent, the shell SHOULD reject or warn (NIP-5D section "Manifest and NUB Negotiation")
- At runtime, `shell.supports()` lets the napplet double-check or query optional capabilities

3.5 **services.has() vs shell.supports()** -- Clarify the distinction:
- `shell.supports('relay')` -- answers "does the shell implement the relay NUB?"
- `services.has('audio')` -- answers "is there an audio service handler registered?"
- NUBs are protocol-level capabilities (relay, signer, storage, ifc, theme)
- Services are optional extensions registered via ServiceRegistry (audio, notifications)
- The shim's services.has() currently uses kind 29010 service discovery round-trip; under NIP-5D it should be synchronous (service list injected at creation or queried via a lightweight envelope)

3.6 **Injection Mechanism** -- Cross-reference Section 2: the shell.supports() capability set is injected using the same mechanism as window.nostr -- at iframe creation time. Document the unified injection approach.

**Section 2: window.nostr Injection (SH-02)**

Write a complete section covering:

2.1 **Background** -- Under RUNTIME-SPEC v2.0.0, napplets accessed signing through the signer proxy: kind 29001 request events over postMessage, kind 29002 response events back. The `@napplet/shim` installed a `window.nostr` object that internally used this round-trip mechanism. The napplet never had a "real" NIP-07 interface -- it had a proxy that looked like NIP-07 but used postMessage under the hood.

NIP-5D changes this: "Shells MUST provide a NIP-07 window.nostr implementation to each napplet iframe." This is a shell responsibility, not a shim responsibility. The shim still installs its own `window.nostr` (for the postMessage-based signer.* NUB), but the shell must ALSO provide a NIP-07 implementation. Document the implications: napplets that don't use `@napplet/shim` (e.g., a vanilla JS napplet that only uses window.nostr) must still have signing capability.

2.2 **The Sandbox Constraint** -- This is the core challenge. Napplet iframes use `sandbox="allow-scripts"` WITHOUT `allow-same-origin`. This means:
- The iframe has an opaque origin -- it cannot access the parent's window.nostr
- The shell cannot directly set `iframe.contentWindow.nostr = ...` because cross-origin iframes restrict property access
- The shell cannot use `iframe.contentDocument` to inject a script because the iframe's document is cross-origin
- Standard approaches (direct property injection, script tag injection via DOM) do NOT work

Document each approach that does NOT work and why, to make the constraint clear.

2.3 **Design Options** -- Present viable injection mechanisms:

**Option A: srcdoc with injected script**
- The shell constructs the iframe's HTML content using the `srcdoc` attribute instead of `src`
- The shell injects a `<script>` block at the top of the srcdoc that defines `window.nostr`
- The injected script uses postMessage to proxy signing requests back to the shell (same mechanism as the shim, but shell-controlled)
- **Pro:** Shell controls the injection; works with sandbox="allow-scripts"
- **Pro:** The nostr object is available synchronously before any napplet code runs
- **Con:** Requires the shell to fetch napplet HTML and inject the script -- changes the iframe loading model from `src` URL to `srcdoc` blob
- **Con:** Large napplets may have issues with srcdoc size limits in some browsers

**Option B: postMessage-based initialization handshake**
- The napplet loads normally via `src` URL
- The shim (or a minimal bootstrap script included in the napplet build) sends a `{ type: "shell.ready" }` message on load
- The shell responds with `{ type: "shell.init", capabilities: {...}, nostrProxy: true }` -- this tells the shim that window.nostr requests should use the signer.* NUB envelope messages
- **Pro:** No change to iframe loading model
- **Pro:** Works with any sandbox configuration
- **Con:** window.nostr is NOT available synchronously -- napplet code that calls `window.nostr.getPublicKey()` at module load time will fail
- **Con:** Requires the napplet to include the shim (vanilla JS napplets without shim have no window.nostr)

**Option C: Shim-as-shell-provided-bootstrap**
- The shell injects the `@napplet/shim` script URL into the napplet's HTML via srcdoc or a Service Worker interception
- The shim itself provides window.nostr via the signer.* NUB postMessage proxy
- The shell's responsibility is ensuring the shim is loaded, not providing a separate window.nostr
- **Pro:** Single implementation of window.nostr (the shim)
- **Pro:** The shim already handles all NIP-07 methods via signer.* NUB
- **Con:** Still requires srcdoc or Service Worker for injection
- **Con:** Couples the shell to the shim implementation

2.4 **Recommended Approach** -- Analyze the options and recommend one, explaining the rationale. Consider:
- The NIP-5D spec says "Shells MUST provide" -- this implies shell responsibility, not napplet/shim responsibility
- The sandbox constraint means the shell cannot directly inject properties on the iframe's window
- The shim already provides a working window.nostr via postMessage -- the question is whether the shell needs a SEPARATE implementation or just needs to ensure the shim's implementation works
- The signer.* NUB messages are the transport -- whether the shell "provides" window.nostr by ensuring the shim loads and routing signer.* messages, or by injecting its own proxy script, the end result is the same: napplet code can call window.nostr.signEvent() and get a result

2.5 **NIP-07 Method Coverage** -- Document which NIP-07 methods must be supported:
- `getPublicKey()` -- returns the user's public key
- `signEvent(event)` -- signs a nostr event
- `getRelays()` -- returns preferred relay URLs
- `nip04.encrypt(pubkey, plaintext)` / `nip04.decrypt(pubkey, ciphertext)` -- NIP-04 DM encryption
- `nip44.encrypt(pubkey, plaintext)` / `nip44.decrypt(pubkey, ciphertext)` -- NIP-44 versioned encryption
Map each to the corresponding signer.* NUB message type from the NUB-signer spec.

2.6 **Security Boundaries** -- Document security concerns:
- The shell controls which iframe gets which signer -- the shell's AuthHooks.getSigner() returns the NIP-07 signer instance
- ACL enforcement: signer operations require capabilities (sign:event, sign:nip04, sign:nip44) per the ACL system
- The window.nostr proxy must go through the runtime's ACL gate, not bypass it
- Consent gating: destructive kinds (0, 3, 5, 10002) still require user consent via registerConsentHandler
- The shell should NOT inject the raw signer private key into the iframe -- all signing is proxied through the shell

2.7 **Affected Files** -- Table:
| File | Change | Impact |
With: shell-bridge.ts or new module (injection logic), hooks-adapter.ts (AuthAdapter usage), types.ts (new hook types if needed)
