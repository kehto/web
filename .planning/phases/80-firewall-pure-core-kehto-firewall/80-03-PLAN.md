## Pinned decisions (RESEARCH A1 + A2 — deterministic, MUST be encoded as documented constants/comments in evaluate.ts and proven in tests)

- **A1 — intra-tier precedence order (POLICY-03):** first-match-wins, most-to-least specific:
  1. per-napplet policy (`allow` short-circuits to pass/ignore; `deny` → reject/block; `ask` → prompt)
  2. init-burst guard (`initElapsedMs` within burst window AND burst counter exceeds `maxOps`) → its action (default `block`)
  3. content matchers (first matching matcher wins) → its action
  4. per-napplet × op-class rate limit (token bucket) → exceed-action
  5. per-napplet global rate fallback (RATE-03) → exceed-action
  6. global default rate (defaults.ts, CORE-04) → exceed-action
  Document this exact order as a numbered comment block at the top of evaluate(), and write a precedence test where a more-specific rule fires over a less-specific one for the same observation.
- **A2 — unfocusedMultiplier scaling (FOCUS-02):** the multiplier scales effective bucket **CAPACITY** (tightens burst headroom) when `observation.focused === false`. Refill rate is derived as `effectiveCapacity / windowMs` so the drip also tightens proportionally; the bucket KEY stays stable (`napplet:opClass`, no focus suffix). Because the multiplier is always `> 0`, an unfocused napplet's budget is reduced but never zero — focus alone NEVER hard-blocks. Document this in evaluate()'s JSDoc and prove it with a focused-vs-unfocused test pair.
</objective>
