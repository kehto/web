# Quick Task 260624-1i5: Resolve kehto/web#88 createBleService host event emission

## Context

Issue #88 reports that `createBleService()` passes only `windowId` to host BLE hooks. NAP-BLE already defines runtime-pushed `ble.event` messages, so host notification subscriptions need a supported way to emit those messages through the reference service.

The local `createWebrtcService()` implementation already uses the target pattern: hook context includes `emit(event)`, and the service sends `{ type: 'webrtc.event', event }` to the requesting napplet.

## Plan

1. Add a failing regression test in `packages/services/src/ble-service.test.ts` proving a `subscribe` hook can call `context.emit({ type: 'notification', ... })` and the service sends `{ type: 'ble.event', event }` without disrupting the `ble.subscribe.result` response.

2. Update `packages/services/src/ble-service.ts` to import `BleEvent`, add `emit(event: BleEvent)` to `BleServiceContext`, and construct the context from the current `send` callback.

3. Update service docs/README examples to show `ctx.emit(...)` for BLE subscriptions, then verify with focused unit tests, package build/type-check, docs check if docs changed, and repo status/diff checks.

## Upstream NAP-BLE Decision

No `napplet/naps` issue is expected unless implementation reveals a missing or contradictory NAP-BLE event contract. Current installed `@napplet/nap@0.21.0` already includes `BleEventMessage` with `type: 'ble.event'` and `event: BleEvent`.
