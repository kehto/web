# Quick Task 260523-h64 Summary

## Goal

Use published npm `@napplet/*` packages instead of repo-local `napplet` and `nubs` submodules, while preserving the playground's hosted NIP-5D/NIP-5A gateway behavior.

## Result

- Removed the `napplet` and `nubs` submodule gitlinks and local package overrides in commit `32cc987`.
- Kept active package resolution on published `@napplet/*@0.3.0` packages.
- Removed local Vite/Vitest aliases that pointed at repo-local `napplet/packages/*` sources.
- Added guards that reject reintroduced local `@napplet/*` overrides, workspace entries, or lockfile links.
- Kept Kehto's playground-specific single-file artifact bridge at the gateway boundary because the published `@napplet/vite-plugin@0.3.0` dist accepts `artifactMode: "single-file"` but does not rewrite final artifacts itself.
- Added a hosted artifact bootstrap so sandboxed napplets post `shell.ready`, consume `shell.init`, and expose shell-derived `window.napplet.shell.supports()` without depending on local shim source.
- Bound playground napplet identity from the gateway-derived iframe/session tuple instead of waiting for a startup envelope, so valid hosted napplets that do not send an initial message no longer remain pending.

## Browser Regression

The reported `postMessage` error and `Error restoring session : Transport request timed out` were reproduced as the same failure class guarded by `tests/e2e/demo-boot.spec.ts`.

Root cause: the branch had moved to published `@napplet/shim@0.3.0`, whose dist does not include the hosted playground `shell.ready`/`shell.init` readiness bridge. Sandboxed gateway iframes have opaque origins, so a sender targeting `http://localhost:5174` can fail when the recipient origin is `null`. The final artifact bootstrap now sends readiness with `"*"`, accepts shell capabilities from `shell.init`, and the E2E guard fails on the exact timeout/origin-error family.
