# Quick Task 260711-r4p: Implement usable NAP-UPLOAD in Kehto Paja

## Objective

Replace Paja's protocol-shaped upload mock with an opt-in real Blossom path while
preserving `memory` as the explicit deterministic simulator. The implementation must
conform to `naps/NAP-UPLOAD.md` at `napplet/naps`
`a7cc17463cbf5d9cb87884b31071bc4fc826034c`: the shell owns policy, server selection,
consent, authorization signing, HTTP egress, and integrity claims.

Use Hyprgate's
`/home/sandwich/Develop/hyprgate/apps/shell/src/lib/services/upload-blossom.ts` and
`apps/shell/src/lib/blossom/blossom-servers.ts` as behavioral guidance for rail
rejection, kind-24242 auth, progress, cancellation, NIP-94 results, and ordered BUD-03
server lists. Do not copy its `blossom-client-sdk` dependency: Kehto already owns the
dependency-free `createHttpUploader` transport.

## Scope decisions

- Keep `memory` as the backwards-compatible default; `blossom` is enabled explicitly
  by config or `--upload-mode blossom`.
- Resolve upload servers in this order: normalized explicit Paja servers, an
  independently warmed/cache-only snapshot of the active signer's newest kind-10063
  ordered `server` tags when discovery is enabled, then fail with
  `no server configured`. Warming belongs to host startup/signer-change lifecycle;
  `upload.upload` and `upload.info` only read the snapshot and never initiate a relay
  query. Never use a public default or runtime-pointer artifact-fetch hints.
- Upload to the first effective server and return its direct HTTP(S) descriptor.
  Multi-server mirroring/failover and BUD-10 URLs are intentionally out of scope and
  must be documented, not implied.
- Accept HTTPS servers everywhere. Accept HTTP only for loopback development hosts
  (`localhost`, `127.0.0.0/8`, and `[::1]`); reject credentials and every non-loopback
  HTTP URL. `upload.info.returns` must describe the first effective server as `https`
  or `http` accordingly, and the returned descriptor must satisfy the same scheme
  policy.
- Require a signer with `signEvent` and one authoritative pubkey. A configured pubkey,
  provider pubkey, signer `getPublicKey()`, discovery cache author, and returned signed
  event pubkey must agree. A configured fixed pubkey is read-only and must fail rather
  than fabricate authorization; mismatched NIP-07/NIP-46 identity also fails.
- Add upload-specific consent before kind-24242 signing or fetch. Do not weaken the
  existing generic signer confirmation boundary; dev/NIP-07/NIP-46 may therefore show
  the existing signing confirmation after the upload disclosure prompt.
