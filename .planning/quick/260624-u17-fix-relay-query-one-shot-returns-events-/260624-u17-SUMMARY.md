# Phase quick-260624-u17 Plan 01: Fix relay.query One-Shot Returns Events Summary

**One-liner:** Rewrote handleRelayQuery to perform one-shot subscribe-until-EOSE via the registered relay service, returning `{ events: NostrEvent[] }` per the canonical RelayQueryResultMessage contract instead of a useless `count`.

## What Was Built

### Task 1 (TDD): Rewrite handleRelayQuery

`handleRelayQuery` in `packages/runtime/src/relay-handler.ts` previously scanned the local event buffer, counted matches, and replied with `{ type: 'relay.query.result', id, count }`. This was non-conformant with the `@napplet/nap` `RelayQueryResultMessage` interface which specifies `events: NostrEvent[]` (no `count` field).

The new implementation:

1. **Buffers matches first** — iterates `eventBuffer.getBufferedEvents()`, collects those passing `matchesAnyFilter`, deduplicating by `event.id` into a `Set<string>`.
2. **Settled flag** — a single `settle()` function guarded by a `settled` boolean sends the `relay.query.result` exactly once and clears any fallback timer.
3. **Shell-kind fast path** — `isShellKindQuery(filters)` true → `settle()` immediately (buffer-only, no relay delegation).
4. **Registered relay service path** — `relayServiceFrom(context)` returns a service; synthesizes `{ type: 'relay.subscribe', subId: '__query__:<id>', filters, relay? }` and calls `svc.handleMessage` with a collector callback. On `relay.eose`, decrements an open-backend counter; when all backends have EOSEd, sends `relay.close` to each and calls `settle()`. A 15-second fallback timer ensures teardown and settlement even if EOSE never arrives.
5. **Runtime-backend fallback** — when no relay service is registered, falls back to `hooks.cache.query()` + `hooks.relayPool.subscribe()`, with EOSE and 15-second timer completing the query.
6. **No `count` field** anywhere in the implementation.

Tests rewritten/added in `dispatch.test.ts`:
- Rewritten: asserts `events` is an array and `count` is undefined
- Rewritten: asserts published event appears in `events` (by id)
- New (a): stub relay-pool service emits events then EOSE; asserts events land in result and `relay.close` is sent
- New (b): same event in buffer and from service appears exactly once in `events` (dedup)

### Task 2: Conformance guard, migration docs, changeset

- `tests/unit/nip5d-conformance-guard.test.ts`: new static guard asserts relay-handler.ts contains `'relay.query.result'`, the result literal carries `events` field, does NOT contain the legacy `'relay.query.result', id, count` shape, and calls `relayServiceFrom(context)`. Also asserts installed `RelayQueryResultMessage` interface field names include `events` and exclude `count`.
- `docs/migrations/RUNTIME-MIGRATION.md`: marked `relay.query.result` count row as **superseded**, added correcting note pointing to issue #94 and canonical `{ events: NostrEvent[] }` contract. ACL mapping unchanged.
