<objective>
Fix issue #94: `relay.query` (one-shot) in `@kehto/runtime` returns a `count` and only
scans the local event buffer, bypassing the registered relay service. Per the canonical
`@napplet/nap` `RelayQueryResultMessage` contract, it must perform a one-shot
subscribe-until-EOSE, delegate to the registered relay/cache service (mirroring
`handleRelaySubscribe`), merge buffered matches, dedupe by `event.id`, and reply once
with `{ type: 'relay.query.result', id, events }`.

Purpose: A relay-backed napplet doing a one-shot query currently gets a useless number
and never sees events from the relay service. This restores wire conformance.

Output: Corrected `handleRelayQuery`, updated + new runtime tests, a conformance guard,
corrected historical migration docs, and a `@kehto/runtime` changeset.
</objective>

```ts
interface RelayQueryResultMessage extends RelayMessage {
  type: 'relay.query.result';
  id: string;            // correlation id matching the request
  events: NostrEvent[];  // matching events (NOT count)
  error?: string;
}
```

Relay service response shapes (packages/services/src/relay-pool-service.ts L127-190):
`{ type: 'relay.event', subId, event }` per event, then `{ type: 'relay.eose', subId }`.
A `{ type: 'relay.close', subId }` tears the subscription down.

- `matchesAnyFilter(event, filters)` over `context.eventBuffer.getBufferedEvents()`

  <files>packages/runtime/src/relay-handler.ts, packages/runtime/src/dispatch.test.ts</files>
  <behavior>
    - relay.query with no registered service and no relay pool replies immediately with
      relay.query.result whose `events` is an array (no `count` field) — buffered matches only.
    - After a relay.publish of a kind-1 event, a subsequent kind-1 relay.query result `events`
      contains that event (assert by id / non-empty length), not `count > 0`.
    - relay.query with a registered relay-pool service: runtime synthesizes a relay.subscribe
      (unique subId, carries filters + optional relay hint) to the service; events the service
      emits via relay.event appear in the result; on relay.eose the runtime sends a relay.close
      to the service to tear it down, then replies once.
    - Dedup: an event present both in the buffer and emitted by the service appears exactly once
      in `events` (deduped by event.id).
    - Single reply only (settled flag guards against EOSE + timeout double-reply).
  </behavior>
  <action>
    Replace the body of `handleRelayQuery` (currently relay-handler.ts ~L330-342). Implement a
    one-shot query that mirrors `handleRelaySubscribe`'s backend selection:

    1. Build the result set: start from buffered matches — iterate
       `context.eventBuffer.getBufferedEvents()`, keep those passing `matchesAnyFilter(event, filters)`.
       Track a `Set<string>` of seen `event.id` for dedup; a collected `NostrEvent[]`.
    2. Define a single `settle()` that, guarded by a `settled` boolean, sends
       `{ type: 'relay.query.result', id, events }` exactly once and clears any fallback timer.
    3. `isShellKindQuery(filters)` true => buffer-only: call `settle()` immediately, return.
    4. `relayService = relayServiceFrom(context)`;
       `cacheService = !serviceRegistry['relay'] ? serviceRegistry['cache'] : undefined`.
    5. Relay/cache service path (relayService present): synthesize an internal subscribe message
       `{ type: 'relay.subscribe', id, subId: '__query__:' + id, filters, relay: m.relay }`
       (only include `relay` when m.relay is a non-empty string). Track how many delegated backends
       are still open (relayService, plus cacheService if present). For each backend call
       `backend.handleMessage(windowId, subscribeMsg, collector)` where the collector: on
       `relay.event` adds `resp.event` to the result set if its id is unseen; on `relay.eose`
       decrements the open-backend count and, when it reaches 0, sends a synthetic
       `{ type: 'relay.close', subId }` to each delegated backend and calls `settle()`. Arm a
       ~15s fallback timer that, if it fires first, sends relay.close to the backends and settles.
    6. Runtime-backend fallback (no relayService): mirror `deliverFromRuntimeBackends`.
       Query `hooks.cache` if `hooks.cache?.isAvailable()` (collect its events). Subscribe via
       `hooks.relayPool` if `hooks.relayPool?.isAvailable()` (selectRelayTier or relay hint),
       collecting events and finishing on pool EOSE; otherwise finish via the fallback timer.
       Untrack/unsubscribe the pool subscription before settling. If neither cache nor pool is
       available, call `settle()` immediately (buffered events only).

    Do NOT emit a `count` field anywhere. Reuse existing helpers; do not duplicate
    `relayServiceFrom` / `isShellKindQuery` / `matchesAnyFilter`. Keep the `setTimeout`/`clearTimeout`
    ambient declarations already at the top of the file.

    In dispatch.test.ts (~L260-298): rewrite the two existing relay.query tests to assert `events`
    (array; contains the published event after a publish) instead of `count`. Add two new tests:
    (a) a stub relay-pool service registered via `runtime.registerService('relay-pool', stub)` whose
    handleMessage emits relay.event(s) then relay.eose for a relay.subscribe and records relay.close
    — assert the emitted events land in the result AND a relay.close was received; (b) dedup — same
    event in the buffer and from the stub service appears once in `events`.
  </action>
