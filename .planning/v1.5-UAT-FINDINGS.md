# v1.5 Input: UAT Findings from v1.4 Demo

## Context

After `v1.4` tag was pushed, user manually smoke-tested the demo in a browser and identified 6 issues spanning AUTH concurrency, shell-UI state aggregation, and sequence-diagram coverage. These bugs passed all v1.4 automated checks (447 unit tests, 49 E2E specs, integration audit) because:

- Playwright specs load **one napplet at a time** (demoBeforeEach loads a single frame under test). The full concurrent boot of all 10 napplets was never exercised in CI.
- `demo-boot.spec.ts` asserts stub-badge counts but doesn't verify per-napplet AUTH success.
- Shell-UI state surfaces (ACL matrix, activity counters, sequence-diagram lanes) had no assertions tying them to NUB envelope traffic.
- Integration checker reviewed code wiring statically but did not run the demo in a browser.

## Issues (6)

### Issue 1: 7/10 napplets stuck on LOADING under concurrent boot [BLOCKER]

**Observed:** Demo boot at `pnpm --filter @kehto/demo dev` shows:
- `bot` ✓ AUTHENTICATED
- `chat` ✓ AUTHENTICATED
- `composer` ✗ stuck on `LOADING…`
- `feed` ✗ stuck on `LOADING…`
- `hotkey-chord` ✗ stuck on `LOADING…` (v1.4 Phase 26)
- `media-controller` ✗ stuck on `LOADING…` (v1.4 Phase 27)
- `preferences` ✗ stuck on `LOADING…`
- `profile-viewer` ✗ stuck on `LOADING…`
- `theme-switcher` ✗ stuck on `LOADING…` (despite theme fan-out working — see issue 3)

**Why Playwright missed it:** Every Layer-B spec loads one napplet frame in isolation via `__loadNapplet__` after `demoBeforeEach`. The "all 10 napplets mount concurrently from topology.ts" boot path was never exercised.

**Likely root cause:** Shell AUTH flow under concurrent startup. v1.3 had 8 napplets working; v1.4 adds 2 (10 total). Possibly iframe contention, AUTH-reply routing collision, or sessionRegistry race condition at 10-way parallel startup.

### Issue 2: Media Play/Pause buttons do nothing [BLOCKER — MEDIA-03]

**Observed:** Clicked Play button in media-controller napplet repeatedly. No state change. `navigator.mediaSession.playbackState` stays untouched (implied by issue 3 activity counter).

**Why E2E-13 passed but this fails:** E2E-13 runs with a single napplet loaded via `__loadNapplet__`, which may bypass the concurrent-boot AUTH failure. In full demo, media-controller never reaches AUTHENTICATED (issue 1), so its play/pause buttons noop because the napplet never gets its session-ready state.

**Likely relation:** Directly downstream of issue 1 — if media-controller doesn't auth, `mediaCreateSession` never fires, so play/pause buttons have no session to report state against.

### Issue 3: Service activity counters stuck at "ACTIVITY: 0 recent" [VISUAL BUG]

**Observed:** Theme service node shows `LAST ACTION: —` and `ACTIVITY: 0 recent` even though user reports theme changes work (theme-switcher is the only v1.3 napplet working — but its activity isn't tracked).

**Scope:** Likely affects ALL service nodes (`identity`, `ifc`, `notify`, `relay`, `storage`, `theme`, `keys`, `media`). Activity counter never ticks on NUB traffic.

**Why Playwright missed it:** No spec asserts service-level activity counters increment on NUB envelope traffic. This was a pre-existing v1.3 shell-UI wiring gap that v1.4 didn't address.

### Issue 4: ACL Capability Matrix shows "No authenticated napplets" [VISUAL BUG]

**Observed:** ACL Capability Matrix modal displays service toggles (identity/keys/media/notifications/relay/signer/storage/theme — all green) and capability column headers (RELAY SUBSCR… / RELAY PUBLIS… / CACHE READ / CACHE WRITE / HOTKEY FORWA… / STATE READ / STATE WRITE / IDENTITY READ / KEYS BIND / KEYS FORWARD / MEDIA CONTROL / NOTIFY SEND / NOTIFY CHANN… / THEME READ / BLOCKED), but body says "No authenticated napplets" despite bot + chat being AUTHENTICATED (issue 1 screenshot).

**Why Playwright missed it:** No spec asserts ACL panel populates with authenticated napplets.

**Likely root cause:** Shell UI ACL-matrix component queries `runtime.sessionRegistry` or similar authenticated-napplet lookup; the lookup is broken or the subscription isn't live-updating.

### Issue 5: Sequence Diagram only shows Chat/Shell/Bot lanes [VISUAL BUG]

**Observed:** Debugger sequence diagram renders three lanes: `Chat`, `Shell`, `Bot`. The other 8 napplets (including bot and chat's siblings) have no lane. Chat shows 18+ rapid `storage.get` / `storage.get.result` pairs, then `ifc.subscribe`, then `relay.subscribe` + several `relay.event` + `relay.eose`.

**Likely root cause:** Debugger tap's lane-generation only tracks napplets that reach AUTHENTICATED (consistent with issue 1 — other napplets never auth so never register a lane). Once issue 1 is fixed, lanes for those napplets should appear.

### Issue 6: Chat makes 18+ `storage.get` serial round-trips at boot [PERF SMELL]

**Observed:** In the Chat sequence diagram, 18+ `storage.get` → `storage.get.result` pairs fire before `ifc.subscribe` + `relay.subscribe`.

**Likely cause:** Chat napplet reading many keys at boot (contacts, rooms, prefs, etc.) via awaited `storage.get` serialization instead of a single batched read or parallel `Promise.all`. Not a correctness bug but a perf-debt item.

## Suggested v1.5 Scope

Primary theme: **Demo Stability & UAT Coverage**

1. **Root-cause + fix concurrent-boot AUTH failure** (issue 1 → cascades to 2, possibly 5)
2. **Shell UI state wiring for activity counters + ACL matrix + sequence-diagram lanes** (issues 3, 4, 5)
3. **CI coverage for multi-napplet concurrent boot** — new Playwright spec loads all 10 napplets against `:4174` and asserts each reaches AUTHENTICATED within a timeout.
4. **CI coverage for shell-UI state surfaces** — specs assert service-activity counters tick, ACL matrix populates, sequence-diagram lanes render per authenticated napplet.
5. **(Optional) Storage batching** for chat boot path (issue 6).
