# Phase 53: Single-File Artifact Contract - Context

**Gathered:** 2026-05-22
**Status:** Ready for execution
**Mode:** Smart discuss skipped; infrastructure/API contract with explicit roadmap boundary

<domain>
## Phase Boundary

Make `@napplet/vite-plugin` intentionally support a gateway-portable single-file NIP-5A artifact. This phase owns the plugin contract only: explicit mode selection, mode-aware inline executable script validation, final-artifact aggregate hash computation, manifest emission, config/connect synthetic hash preservation, direct tests, and public package docs/types.

</domain>

<decisions>
## Implementation Decisions

### Artifact Mode Contract
- Add an explicit single-file artifact option to `Nip5aManifestOptions`; do not silently infer the mode from Vite output shape.
- Keep existing external-asset behavior guarded by default, so accidental inline executable scripts still fail unless single-file mode is opted in.
- Accept only build-produced inline executable scripts in explicit single-file mode; non-executing script/data tags remain allowed as before.
- Compute the aggregate hash from the final emitted artifact bytes after any inlining and before manifest/meta update bytes are written.

### Hash and Manifest Semantics
- Preserve config schema and connect-origin synthetic xTag inputs in single-file mode.
- Keep synthetic xTags out of emitted `['x', ...]` manifest projection while retaining their aggregateHash influence.
- Treat the served artifact as the source byte boundary for gateway parity: in single-file mode the primary real `x` tag should be the final `index.html` artifact rather than pre-inline JS/CSS chunks.
- Avoid package-wide dependency changes; use Vite/Rollup hooks already available to the plugin.

### Testing and Docs
- Add direct `@napplet/vite-plugin` unit tests because the package currently has no direct tests for this contract.
- Tests should cover default inline-script rejection, explicit single-file acceptance, final artifact hash calculation, and synthetic config/connect hash participation.
- Update README/type surface and package test script so future plugin changes run through the root unit test loop.

### the agent's Discretion
- Choose the smallest internal helper extraction needed to make behavior testable without exposing extra public API.
- Prefer deterministic test fixtures over invoking a full Vite build where the plugin lifecycle can be tested directly.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `napplet/packages/vite-plugin/src/index.ts` contains the public `nip5aManifest()` factory, `Nip5aManifestOptions`, inline-script guard, aggregate hash helper, manifest assembly, and meta injection path.
- `SYNTHETIC_XTAG_PATHS` already centralizes synthetic hash inputs (`config:schema`, `connect:origins`) for aggregateHash inclusion but manifest xTag exclusion.
- `napplet/packages/nub/src/connect/aggregate-hash.test.ts` documents connect-origin fold expectations that the plugin already mirrors.
- Root `vitest.config.ts` already includes `tests/unit/**/*.test.ts`; `napplet/vitest.config.ts` includes `packages/*/src/**/*.test.ts`.

### Established Patterns
- GSD phase execution keeps one plan summary and one verification artifact in `.planning/phases/<phase>/`.
- Package tests use Vitest and focused package-level `pnpm --filter <pkg> type-check` commands before full milestone verification.
- Plugin dist/type output is generated by `tsup`; source and README are the tracked public contract surfaces.

### Integration Points
- `transformIndexHtml()` injects placeholder meta tags.
- `closeBundle()` reads `dist/index.html`, validates scripts, computes file hashes, adds synthetic config/connect hash inputs, writes `.nip5a-manifest.json`, and replaces the aggregate-hash placeholder.
- Playground napplet `vite.config.ts` files consume `nip5aManifest()` and will opt into the explicit single-file mode in Phase 54.

</code_context>

<specifics>
## Specific Ideas

- Name the new option `artifactMode: 'external-assets' | 'single-file'` with default `'external-assets'`.
- Add a test fixture helper that creates a temporary `dist/index.html`, invokes the plugin lifecycle hooks, and reads emitted HTML/manifest output.
- Use a deterministic `VITE_DEV_PRIVKEY_HEX` in tests so manifest generation and meta replacement run.
- Hash assertions should independently compute `sha256(index.html-before-meta-replacement)` and compare it against the plugin's manifest/meta aggregate hash.

</specifics>
