From packages/acl/src/check.ts:
```typescript
export function toKey(identity: Identity): string {
  return `${identity.pubkey}:${identity.dTag}:${identity.hash}`;
}

export function check(state: AclState, identity: Identity, cap: number): boolean {
  const key = toKey(identity);
  const entry = state.entries[key];
  if (!entry) return state.defaultPolicy === 'permissive';
  if (entry.blocked) return false;
  return (entry.caps & cap) !== 0;
}
```

**Section 3: Persisted ACL Data Migration (ACL-03)**

Write a section titled `## 3. Persisted ACL Data Migration` that contains:

3.1. An opening paragraph explaining: @kehto/shell persists ACL state via `serialize()` to localStorage (key: `napplet:acl`). The serialized format is `JSON.stringify(aclState)` where `aclState.entries` is keyed by composite strings. After the identity key schema change (Section 1), old persisted entries under `pubkey:dTag:hash` keys become orphaned — `toKey()` now produces `dTag:hash`, so lookups against old entries always miss. A one-time migration is required.

3.2. A **subsection** `### Persisted Data Format` showing the old and new JSON formats:

Old format example:
```json
{
  "defaultPolicy": "permissive",
  "entries": {
    "3a1b2c...f9:chat:e3b0c4...42": { "caps": 33, "blocked": false, "quota": 524288 },
    "7d8e9f...a1:notes:ab12cd...ef": { "caps": 1023, "blocked": false, "quota": 524288 }
  }
}
```

New format example:
```json
{
  "defaultPolicy": "permissive",
  "entries": {
    "chat:e3b0c4...42": { "caps": 33, "blocked": false, "quota": 524288 },
    "notes:ab12cd...ef": { "caps": 1023, "blocked": false, "quota": 524288 }
  }
```

3.3. A **subsection** `### Migration Strategy` with numbered steps:

**Step 1: Detect old-format entries.** On shell startup (or first ACL load after migration), deserialize the stored ACL state. Check if any entry key matches the old 3-segment pattern `*:*:*` (contains exactly two colons). If no 3-segment keys exist, skip migration.

**Step 2: Extract dTag:hash from old keys.** For each old-format key (`pubkey:dTag:hash`), split on `:` and take the last two segments: `dTag` and `hash`. Reconstruct as `dTag:hash`. Note: this assumes dTag and hash never contain `:` characters — document this as a precondition (which holds today since dTag is a NIP-5A d-tag identifier and hash is hex).

**Step 3: Merge entries.** For each extracted `dTag:hash` key:
- If no entry exists under the new key: copy the old entry as-is
- If an entry already exists under the new key (from a NIP-5D session): merge by OR-ing caps bitfields (`newEntry.caps | oldEntry.caps`), keeping `blocked: true` if either is blocked, and keeping `max(newEntry.quota, oldEntry.quota)`

**Step 4: Remove old-format keys.** Delete all entries with 3-segment keys from the entries record.

**Step 5: Persist.** Serialize and write the migrated state back to localStorage.

3.4. A **subsection** `### Migration Utility (Pseudocode)` with a TypeScript function:
```typescript
function migrateAclState(state: AclState): AclState {
  const newEntries: Record<string, AclEntry> = {};
  let migrated = false;

  for (const [key, entry] of Object.entries(state.entries)) {
    const parts = key.split(':');
    if (parts.length === 3) {
      // Old format: pubkey:dTag:hash -> dTag:hash
      const newKey = `${parts[1]}:${parts[2]}`;
      const existing = newEntries[newKey];
      if (existing) {
        // Merge: union caps, preserve block, max quota
        newEntries[newKey] = {
          caps: existing.caps | entry.caps,
          blocked: existing.blocked || entry.blocked,
          quota: Math.max(existing.quota, entry.quota),
        };
      } else {
        newEntries[newKey] = entry;
      }
      migrated = true;
    } else {
      // Already new format or other key — preserve
      newEntries[key] = entry;
    }
  }

  if (!migrated) return state; // No old entries found

  return { defaultPolicy: state.defaultPolicy, entries: newEntries };
}
```

3.5. A **subsection** `### Rollback Considerations`:
- **Pre-migration backup**: Before running migration, copy the raw localStorage value to a backup key (e.g., `napplet:acl:backup-v2`). This is a single `localStorage.getItem` + `localStorage.setItem`.
- **Rollback procedure**: If migration causes issues, restore from backup key. This reverts to old-format entries, which only work with a non-migrated `toKey()`. Rollback requires both restoring the ACL data AND reverting the `toKey()` function — partial rollback breaks ACL entirely.
- **Idempotency**: The migration function is idempotent — running it on already-migrated data (no 3-segment keys) returns the state unchanged.
- **Data loss risk**: LOW. The only data loss scenario is if two different pubkeys had entries for the same `dTag:hash` combination with conflicting `blocked` status. The merge strategy (OR caps, OR blocked, MAX quota) is conservative — it never removes capabilities, and it blocks if any source was blocked. Document this as intentional: security-conservative merge.
- **Forward compatibility**: After migration, the `deserialize()` function in @kehto/acl already validates entry structure without checking key format. Old and new key formats are both valid strings. No changes to `deserialize()` are needed.

3.6. A **subsection** `### Where Migration Runs`:
- The migration utility belongs in `@kehto/shell` (not `@kehto/acl`), because `@kehto/acl` is a pure, zero-deps module with no I/O. The shell's `acl-store.ts` handles localStorage persistence and is the natural location.
- Migration triggers on `aclStore.load()` — after deserializing, run `migrateAclState()`, then persist if changed.
- Migration runs exactly once per browser context — subsequent loads find no 3-segment keys and skip.
