<objective>
Implement the RENAME-02 soft-rename for the shell-bridge identity-changed topic: any caller that invokes `bridge.injectEvent('auth:identity-changed', payload)` OR `bridge.injectEvent('identity:changed', payload)` triggers a dual-emit — both topics are forwarded to `runtime.injectEvent` so subscribers of either topic name continue to receive the event for one release. All non-identity topics are forwarded exactly once with the topic string unchanged. JSDoc, changeset, PROJECT.md Known Tech Debt, and a new unit test lock the behavior. Hard-removal of the dual-emit is scheduled for v1.9.

Purpose: Clear the v1.6 carryover identified as a "live surface with external consumers" — the soft-rename style protects downstream hosts (e.g. hyprgate) that subscribe to the old `'auth:identity-changed'` topic from breaking on the v1.8 upgrade, while announcing the canonical new name `'identity:changed'` (matching the NIP-5D `identity` NUB domain naming convention).
