<dual_registration_rationale>
Why dual-register instead of renaming?

1. **Topology display** (apps/demo/src/shell-host.ts:106-114) lists `'notifications'` in `DEMO_TOPOLOGY_SERVICE_NAMES`. Renaming would force every topology consumer (node-inspector, flow-animator, color-state, debugger) to switch to `'notify'`, breaking the v1.2 conformance tests.

2. **Host-originated wiring** (apps/demo/src/notification-demo.ts:147-184) calls `_serviceHandler.handleMessage(...)` directly with envelopes addressed via the explicit handler reference, NOT via runtime dispatch. It uses `getNotificationServiceHandler()` (apps/demo/src/shell-host.ts:543) which returns `_notificationServiceHandler` — also independent of the registry name.

3. **Runtime envelope routing** (packages/runtime/src/runtime.ts:1000) keys on the literal string `'notify'` for `notify.*` envelope dispatch. Adding a second registration entry under `'notify'` referencing the same handler instance is the minimal change.

4. **Topology node count** stays at 8 (no new node added — `'notify'` is not in DEMO_TOPOLOGY_SERVICE_NAMES, so it doesn't render a new topology card). Phase 17 contract (8 service nodes) preserved.

5. **No protocol change.** Same service handler, same envelope contract. Just two strings pointing at the same instance.

This is consistent with Phase 17 decision (Phase 17-demo-app-rewire) "demoServiceNames seeded with 8 domains (including topology-only relay/storage/signer) so topology renders 8 nodes without requiring ServiceHandler registration for those 3" — registry membership is decoupled from topology display.
</dual_registration_rationale>
