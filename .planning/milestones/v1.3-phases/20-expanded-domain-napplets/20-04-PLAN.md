<objective>
Create the `theme-switcher` demo napplet (NAP-08) that lets the user pick a theme and broadcasts it to the demo host. The demo host (Plan 20-06) listens for `{ type: 'demo.publishTheme' }` postMessages from this napplet and forwards to `relay.publishTheme(theme)` (the ShellBridge method that fan-outs `theme.changed` to every napplet via session-registry).

Per CONTEXT D-04: SDK does NOT expose `theme.publish` (verified against /home/sandwich/Develop/napplet/packages/sdk/src/index.ts — there is no `theme` namespace export, only the type re-exports). So this napplet uses ONE narrowly-scoped `window.parent.postMessage` call inside each click handler — this mirrors Plan 19-03's documented SDK gap exemption pattern from the toaster napplet.

CRITICAL ANTI-FEATURE NOTE: theme-switcher is NOT exempt from the `window.addEventListener('message')` ban — it OUTBOUND postMessages only. There is no inbound listener in this napplet. Plan 19-07's ban (0 occurrences allowed for non-toaster napplets) applies here.

Output: `apps/demo/napplets/theme-switcher/` directory, 5-file skeleton.

Plan 20-06 will install a host-side listener at apps/demo/src/main.ts that:
  - Listens for window message events with `event.source` matching one of the napplet iframes
  - Filters for `event.data.type === 'demo.publishTheme'`
  - Calls `relay.publishTheme(event.data.theme)` (where relay is the ShellBridge from shell-host.ts)
  - The ShellBridge.publishTheme() then fan-outs `theme.changed` envelopes to every registered napplet via originRegistry.getIframeWindow + win.postMessage(envelope, '*')

This napplet's only obligation is to dispatch the outbound `demo.publishTheme` message correctly — the host wiring is Plan 20-06's responsibility.
