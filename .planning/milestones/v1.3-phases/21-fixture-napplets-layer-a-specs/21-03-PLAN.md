<runtime_behaviors>
Critical runtime behaviors verified against `packages/runtime/src/runtime.ts`:

1. **identity (handleIdentityMessage, line 720+):** When NO 'identity' service is registered, runtime falls back. `getPublicKey` returns `{ type: 'identity.getPublicKey.error', error: 'no signer configured' }` if `hooks.auth.getSigner()` returns null (default mock). The fixture's `await identity.getPublicKey()` will REJECT in that case — caught by try/catch, sentinel becomes `denied:no signer configured`. Spec must tolerate BOTH paths (success with truncated pubkey, OR denial). The mock pubkey from `mockResult.setUserPubkey('0'.repeat(64))` exists but `getSigner()` returns null. Spec should NOT call `__setIdentityPubkey__` or `__setSigner__` — just observe whichever path the runtime takes.

2. **ifc (handleIfcMessage, line 829+):** `ipc.subscribe` adds windowId to `ifcSubscriptions[topic]` and emits `ifc.subscribe.result` immediately. `ipc.emit` delivers `ifc.event` to all subscribers EXCEPT sender. Single-fixture round-trip: spec injects an `ifc.event` envelope to the fixture (the runtime would normally route this from another napplet, but `__injectEnvelope__` lets us simulate it). After injection, the fixture's `ipc.on(TOPIC, callback)` fires, appending to `#nub-ifc-log`.

3. **notify (handleNotifyMessage, line 1011+):** Runtime fallback handles `notify.send` — returns `{ type: 'notify.send.result', id, notificationId: 'shell-<timestamp>' }`. Fixture's `await notify.send({...})` resolves with that id. Spec asserts `#nub-notif-id` matches `/^shell-\d+$/` AND `__getNubMessage__(windowId, 'notify.send')` returns the request envelope.

4. **relay (handleRelayMessage, line 465+):** `relay.publish` delegates to hooks. The harness's `mockHooks` (`tests/helpers/mock-hooks.ts:54`) has a stubbed `relayPool` that does NOT actually publish events — instead `getRelayPool()` returns `mockRelayPool` (an in-memory pool from `mock-relay-pool.ts`). Behavior: the runtime emits the publish envelope, the relay-publish path either succeeds with a signed event or fails because no signer is set (default). Spec tolerates both. The KEY assertion is `__getNubMessage__(windowId, 'relay.publish')` — proves the envelope was DISPATCHED, regardless of publish success.

5. **storage (StorageProxy, hooks-adapter.ts:256):** localStorage-backed. `storage.setItem('nub-storage-key', 'fixture-v1')` writes to `localStorage` keyed by napplet identity. `storage.getItem` reads it back. Round-trip is reliable in the harness (no signer dependency).

6. **theme (handleThemeMessage, line 1037+):** Runtime fallback returns `DEFAULT_THEME` for `theme.get`: `{ type: 'theme.get.result', theme: { colors: { background: '#0a0a0a', text: '#e0e0e0', primary: '#7aa2f7' } } }`. Fixture itself does NOT call theme.get (no SDK namespace). Spec drives `theme.get` via `__injectEnvelope__(windowId, { type: 'theme.get', id: 'spec-1' })` and asserts `__getNubMessage__(windowId, 'theme.get')` returns the request envelope. To assert the result envelope, spec polls until the runtime's reply is captured by the harness's envelope log — but the result envelope is shell→napplet, NOT napplet→shell, so it does NOT land in `envelopeLog` (which only records napplet outbound). Workaround: spec asserts via `__registerService__('theme', ...)` which lets us intercept theme messages at the runtime, OR more simply: spec asserts the request envelope was dispatched and the fixture's `#nub-status` is 'authenticated' (proves the runtime is alive).

<background>

baseURL: Specs target `:4173` (harness). Inherit the default from `playwright.config.ts` — do NOT override with `test.use({ baseURL: '...' })` because the default is already 4173.
