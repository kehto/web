Extend `createKeysService` with a static `reservedChords?: ReadonlyArray<string>` option. When a napplet forwards a chord matching the reserved set via `keys.forward`, the shell's `onForward` callback (or the `hostBridge`-registered handler, whichever applies) fires — but the service MUST NOT dispatch `keys.action` envelopes to any napplet that registered the same chord via `keys.registerAction`. This is the service-layer contract for KEYS-04 + KEYS-05. README docs (KEYS-06) and the E2E spec (E2E-17) land in 33-02 and 33-03 against this surface.

Purpose: Unblocks hyprgate v2.0 (kehto#8) by giving shells a declarative way to claim WM-absolute chords (workspace switch, window kill, launcher) without a napplet inadvertently intercepting them via `keys.registerAction`. The shell WANTS the forward — that's why it reserved the chord.
