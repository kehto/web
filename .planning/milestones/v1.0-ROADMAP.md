# Milestone v1.0: NIP-5D Migration & Gap Analysis

**Status:** ✅ SHIPPED 2026-04-07
**Phases:** 1-5
**Total Plans:** 7

## Overview

Produced five migration/analysis documents covering the complete specification delta between the previous napplet protocol (RUNTIME-SPEC.md, protocol v2.0.0) and NIP-5D v0.1.0. The gap analysis establishes shared boundary contracts; per-package migration documents provide implementation guides for acl, runtime, shell, and services.

## Milestone Summary

**Key Accomplishments:**
1. Complete gap analysis with wire format before/after for 19 NUB message types and 6 silent failure points
2. ACL identity key schema migration design (pubkey:dTag:hash → dTag:hash) with localStorage migration utility
3. NUB domain-prefix dispatch design replacing NIP-01 verb switch, with AUTH removal inventory (~24% of runtime.ts)
4. SessionEntry identity anchor resolved: Option B (empty string + identitySource discriminant)
5. Shell migration: window.nostr injection via postMessage handshake, shell.supports() capability advertisement
6. All 6 service handlers documented with per-handler migration from NIP-01 arrays to NappletMessage envelopes

**Key Decisions:**
- Dedicated `.error` type suffix for error responses (not `.result` with error field)
- `pubkey` field name for signer.getPublicKey.result (nostr convention)
- No ACL required for read-only signer operations (getPublicKey, getRelays)
- Migration order: acl → runtime → shell → services
- Dual-mode dispatch: NUB path first, legacy array fallback

**Issues Deferred:**
- Theme NUB implementation (net-new, no existing kehto infrastructure)
- IFC channel mode (point-to-point messaging, no existing napplets use it)
- Actual code implementation of all migration changes (next milestone)

**Technical Debt:**
- None remaining (7 cross-document inconsistencies identified and resolved during audit)
