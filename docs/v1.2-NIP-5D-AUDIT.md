# NIP-5D Conformance Audit (v1.2)

**Date:** 2026-04-17
**Scope:** Cross-package drift against [`specs/NIP-5D.md`](../specs/NIP-5D.md) and the `@napplet/nub-*` message surface.
**Milestone:** v1.2 — NIP-5D Conformance & Full NUB Coverage
**Pinned spec:** [`specs/NIP-5D.md`](../specs/NIP-5D.md) (synced from `napplet/specs/NIP-5D.md` at v1.2)

## How to read this document

Every drift item carries a stable ID of the form `DRIFT-<AREA>-<NN>` where `<AREA>` is one of
`ACL`, `RT` (runtime), `SHELL`, `SVC` (services), or `CORE` (dispatch / cross-cutting).
Every item has an owning package file, a one-line remediation, and a Target Phase mapping
(Phase 12, 13, or 14) so downstream planners can cite IDs directly.

Table shape for every drift section: six columns — ID, Drift Item, Current State, Spec/Package Requirement, Target Phase, Remediation Note.

## Scope of audit

- NIP-5D spec-level drift (protocol envelope shape, error suffix, dispatch contract).
- `@napplet/nub-*` message surface gaps for `ifc`, `relay`, `signer`, `storage`, `theme`.
- ACL capability-mapping gaps for the full nub surface.
- Dispatch/Core API drift (the hand-rolled `switch (domain)` in `runtime.ts` vs `createDispatch()` from `@napplet/core`).

Summary of every NUB message type by domain, with coverage status across kehto. Detailed drift

Cell conventions:
- Runtime Dispatch: `no` or `yes (packages/runtime/src/<file>.ts:<line>)`.

- ACL Capability: `no` or `yes (<capability-name>)` where the name is the string returned by `resolveCapabilitiesNub` in `packages/acl/src/resolve.ts`.

## @kehto/acl

ACL drift — capability mapping gaps in `packages/acl/src/resolve.ts`. Every NUB message type that
downstream runtime/services will dispatch must have a `resolveCapabilitiesNub` case, otherwise the
default `{null, null}` fallthrough silently disables enforcement.

| ID | Drift Item | Current State | Spec/Package Requirement | Target Phase | Remediation Note |
|----|------------|---------------|--------------------------|--------------|------------------|
| DRIFT-ACL-01 | No capability mapping for `ifc.channel.*` actions (open, open.result, emit, event, broadcast, list, list.result, close, closed) | `packages/acl/src/resolve.ts:120` `case 'ifc':` only checks `action === 'emit'`; every `channel.*` action falls into `relay:read` by default, which is wrong for `channel.open` / `channel.broadcast` (should require write-equivalent) | `@napplet/nub-ifc` exports 9 channel message types (see `IfcChannelMessage` union) — each requires an ACL decision | Phase 12 | Add explicit cases in `resolve.ts` switch for `action?.startsWith('channel.')` mapping `channel.open`/`channel.emit`/`channel.broadcast` → `relay:write`, `channel.list`/`channel.close` → `relay:read`, reply `.result`/`.event`/`.closed` types → `{null,null}` |
| DRIFT-ACL-02 | Storage domain fallthrough treats any unknown `storage.<action>` as `state:write` | `packages/acl/src/resolve.ts:116` maps only `get`/`keys` to read; every other action (including unknown future additions) becomes `state:write` | `@napplet/nub-storage` exports exactly 4 request types (get, set, remove, keys); the audit should deny unknown storage actions rather than grant write | Phase 12 | Replace the else-branch at `resolve.ts:117-119` with an explicit allowlist: `set` / `remove` → `state:write`; unknown → return `{null,null}` and let the dispatch layer reject |
| DRIFT-ACL-03 | Theme domain has no capability bit defined; `theme.*` returns `{null,null}` silently | `packages/acl/src/resolve.ts:124` returns `{senderCap: null, recipientCap: null}` for any `theme.*` action; no `theme:read` bit in `packages/acl/src/types.ts` | `@napplet/nub-theme` is read-only from napplet's perspective — it needs an enforceable capability so shell hosts can deny theme access per-napplet | Phase 13 | Add `CAP_THEME_READ` constant in `packages/acl/src/types.ts` and map `theme.get` → `theme:read` in `resolve.ts`; include `theme:read` in `ALL_CAPABILITIES` |
| DRIFT-ACL-04 | `ifc.subscribe.result` (reply type) would route through `resolveCapabilitiesNub` and match `ifc` fallthrough to `relay:read`, incorrectly gating shell→napplet replies | `packages/acl/src/resolve.ts:120` does not distinguish napplet→shell vs shell→napplet direction; reply envelopes like `*.result` and `ifc.event` should never be ACL-checked | `NappletMessage` envelope types include reply/event variants; ACL should short-circuit for types ending in `.result`, `.error`, `.event`, `.closed`, `.eose` | Phase 12 | Add an early-return at `resolve.ts:103` for types matching `/(\.result\|\.error\|\.event\|\.closed\|\.eose)$/` → `{null,null}` (reply-direction short-circuit) |
